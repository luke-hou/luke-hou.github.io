PM x Gemini CLI (四)：AI 提升 PM 工作效率的進階實踐

發布日期：2025年10月

本篇文章將介紹能貫穿整個產品開發流程的實用 AI 技巧，提升溝通與分析效率。

文章大綱：

  • 要點一：利用 Gemini CLI 協作產生 Mermaid 流程圖
  • 要點二：利用 Gemini CLI 聯網搜尋能力蒐集資料
  • 要點三：建立 todo.md，讓 AI 逐步完成複雜任務
  • 要點四：AI 輔助撰寫驗收測試案例 (Test Case)
  • 要點五：善用 Markdown 檔案管理複雜指令

要點一：用「說」的畫流程圖 (Mermaid 語法)

痛點場景： 為了跟工程師溝通一個複雜的用戶流程，您是否還在用 PowerPoint 或其他繪圖工具，辛苦地拉著線條、對齊著方塊？

現在，您可以換個方式：用「說」的畫流程圖。您只需用簡單的文字描述流程，AI 就能為您生成一種叫做 「Mermaid」的標準化圖表代碼。

範例：請 AI 生成「用戶登入流程圖」

您只要像平常一樣，條列出您的邏輯步驟：

請幫我生成一個 App 用戶登入的 Mermaid 流程圖，步驟如下：
1.  用戶打開 App
2.  檢查是否已登入
3.  如果已登入，就直接進主頁
4.  如果未登入，就顯示登入頁
5.  用戶輸入帳號密碼後點擊登入
6.  系統驗證帳密
7.  如果成功，進主頁
8.  如果失敗，顯示錯誤訊息並留在登入頁

Gemini CLI 就會產出對應的 Mermaid 代碼：

```mermaid
graph TD
    A[用戶打開 App] --> B{檢查是否已登入?};
    B -- 是 --> C[進入主頁];
    B -- 否 --> D[顯示登入頁面];
    D --> E[用戶輸入帳號密碼];
    E --> F[點擊登入按鈕];
    F --> G{驗證帳號密碼?};
    G -- 成功 --> C;
    G -- 失敗 --> H[顯示錯誤訊息];
    H --> D;
```

您可以將這段純文字的代碼，貼到任何支援 Mermaid 的地方 (例如 Notion、HackMD、VS Code 外掛，甚至是公司的 Wiki)，它就會自動變成一張漂亮的流程圖。這代表您的流程圖可以被版本控制、可以被輕易地複製分享，徹底告別過去的截圖地獄。

從此以後，當流程變動時，您再也不用重新拉線對齊，只需修改幾個字，就能產生新的流程圖，大幅提升溝通效率。

要點二：讓 AI 當您的「24 小時情報蒐集員」

痛點場景： 明天就要報告新功能的市場分析，您還在 Google 上手動搜尋、一篇篇點開文章、複製重點到筆記中？

Gemini CLI 內建了強大的聯網搜尋能力，能讓 AI 直接上網找資料，並根據您的要求進行即時的整理、分析、總結。它就是您最得力的情報官。

範例：10 分鐘完成競品功能分析

假設您想知道市場上的主流專案管理工具 (Asana, Jira, Trello) 是如何做「甘特圖」功能的，您可以這樣問：

你現在是一位資深的產品分析師。請幫我分析一下市面上三個主要專案管理工具 Asana, Jira, Trello 的「甘特圖」功能。

請直接上網搜尋最新資料，並從以下三個角度比較它們的優缺點：
1.  **易用性** (對非技術人員是否友善)
2.  **功能深度** (是否支援複雜的專案管理需求)
3.  **整合性** (與自身平台其他功能的結合程度)

最後，請用一個總結表格呈現你的分析。

Gemini CLI 模擬輸出：

好的，正在為您進行競品分析... 
GoogleSearch Searching the web for: "Asana vs Jira vs Trello Gantt chart features comparison 2025"

分析報告如下：

**主流專案管理工具甘特圖功能比較**

| 工具 | 易用性 | 功能深度 | 整合性 |
| :--- | :--- | :--- | :--- |
| **Asana** | ⭐⭐⭐⭐⭐ (非常直觀) | ⭐⭐⭐⭐ (中大型專案適用) | ⭐⭐⭐⭐⭐ (與任務無縫整合) |
| **Jira** | ⭐⭐ (較為複雜) | ⭐⭐⭐⭐⭐ (極度強大，適合開發) | ⭐⭐⭐⭐ (需透過插件整合) |
| **Trello**| ⭐⭐⭐⭐ (簡單好上手) | ⭐⭐ (適合小型專案) | ⭐⭐⭐ (需透過 Power-Up 整合) |

**總結觀察：**
- **Asana** 在易用性和功能深度上取得了最佳平衡，適合大部分混合型團隊。
- **Jira** 功能最專業、最深度，但學習曲線也最陡峭，是軟體開發團隊的首選。
- **Trello** 最輕量、最簡單，但功能也最受限，適合個人或小型團隊的簡單任務管理。

**給您的建議：** 如果您的目標用戶包含大量非技術人員 (如 PM、行銷、設計)，**Asana** 的設計會是最好的參考起點。

過去需要一小時的資料搜集與整理，現在 AI 在幾分鐘內就能完成，讓您能把時間專注在更高層次的策略判斷上。

要點三：用 todo.md 管理你的「AI 開發團隊」

痛點場景： 您是否試過給 AI 一個非常複雜的指令，例如「幫我分析這份50頁的 PDF 市場報告，並做成簡報」，結果 AI 不是當機，就是給出牛頭不對馬嘴的答案？

這是因為大型語言模型就像人類一樣，一次只能專注做好一件事情。與其把 AI 當成一個什麼都懂的魔術師，不如把它當成一個「能力極強、但需要清晰指令的開發者」。而您，就是那個負責拆解任務的 PM。

最佳實踐：用 todo.md 進行敏捷協作

  1. 建立 todo.md：在專案中建立一個任務清單檔案。
  2. 拆解任務：作為 PM，將您的最終目標拆解成一系列具體、可執行的小步驟。
  3. 逐一指派：一次只給 AI 一個指令，對應清單上的一個任務。
  4. 更新進度：AI 完成後，回到 todo.md 將該項任務打勾 [x]，然後再進行下一步。

範例：請 AI 幫您重構一個網頁

您的 todo.md 可能長這樣：

# 重構「用戶個人資料」頁面

- [ ] 讀取並理解 `user-profile.html` 的現有結構。
- [ ] 分析並列出現有的 CSS 樣式有哪些問題。
- [ ] 使用 Bootstrap 5 重新設計一個更現代化的版面，並生成新的 HTML 檔案 `user-profile-new.html`。
- [ ] 為新版面撰寫一段簡要的設計說明，解釋改動的理由。

接著，您可以開始指派第一個任務：

我們開始執行 `todo.md` 的第一個任務。請先讀取 `@user-profile.html` 這個檔案，並告訴我你理解了它的結構。

這種模式能確保您永遠掌握主導權，AI 的每一步都在您的控制之下，最終產出的品質自然會大幅提升。

要點四：讓 AI 自動生成「QA 的驗收清單」

痛點場景： 您辛苦寫完的 PRD，交給 QA 團隊後，他們寫出來的測試案例，總是因為對需求的理解有偏差，而漏掉一些您在意的細節或邊界條件？

現在，您可以讓 AI 扮演 PM 和 QA 之間的橋樑。在您寫完規格後，直接讓 AI 根據您的規格，產出第一版的「驗收測試案例 (Test Case)」。

範例：將「貼文發布」規格轉為測試案例

假設您的規格文件 spec.md 中有一段：

**功能：貼文發布**
- **使用者故事**：作為社群成員，我希望能發布一篇包含標題和內容的貼文。
- **驗收條件**：
    - 標題和內容都是必填。
    - 標題不可超過 100 字。
    - 成功發布後，應跳轉到新貼文頁面。

您可以這樣對 AI 下指令：

你是一位頂尖的 QA 專家。請根據我提供的規格文件 `@spec.md`，為「貼文發布」功能撰寫詳盡的驗收測試案例。

請務必包含：
- **正面案例** (正常操作)
- **負面案例** (錯誤操作)
- **邊界案例** (測試極端條件)

請用表格形式呈現，欄位包含「案例ID、測試項目、步驟、預期結果」。

Gemini CLI 模擬輸出：

**「貼文發布」功能驗收測試案例**

| 案例 ID | 測試項目 | 步驟 | 預期結果 |
| :--- | :--- | :--- | :--- |
| TC-001 | **正面**：成功發布 | 1. 輸入50字標題
2. 輸入200字內容
3. 點擊「發布」 | 1. 貼文成功
2. 頁面跳轉到新貼文 URL |
| TC-002 | **負面**：標題為空 | 1. 不輸入標題
2. 輸入內容
3. 點擊「發布」 | 1. 提示「標題為必填」
2. 停在原頁面 |
| TC-003 | **負面**：內容為空 | 1. 輸入標題
2. 不輸入內容
3. 點擊「發布」 | 1. 提示「內容為必填」
2. 停在原頁面 |
| TC-004 | **邊界**：標題剛好100字 | 1. 輸入100字標題
2. 輸入內容
3. 點擊「發布」 | 1. 貼文成功
2. 頁面跳轉 |
| TC-005 | **邊界**：標題101字 | 1. 輸入101字標題
2. 輸入內容
3. 點擊「發布」 | 1. 提示「標題不可超過100字」
2. 停在原頁面 |

這份由 AI 生成的測試案例，可以作為您和 QA 團隊討論的基礎，確保每個人對需求的理解都在同一個水平上，從源頭提升產品質量。

要點五：用 Markdown 檔案準備「給 AI 的書面簡報」

痛點場景： 當您有一個複雜的想法或任務，想在小小的終端機對話框裡一次打完，是不是常常覺得很難編輯、很容易按錯送出、或是事後很難找回當初完整的指令？

一個專業的工作習慣是：不要在對話框裡寫長篇大論。而是先把您的想法，有條理地寫在一個 Markdown (`.md`) 檔案裡。這對於處理我們在第三篇提到的「生成 PRD 初稿」等複雜任務時，尤其有用。

您可以把這個檔案，看作是一份要交給 AI 助理的「書面工作簡報 (Briefing)」。在裡面用標題、列表、粗體字，把背景資訊、任務要求、輸出格式等都寫清楚。

寫完後，只要在對話框中簡單地說：

請根據我這份簡報 `@my_complex_idea.md` 的內容，完成指定的任務。

這樣做的好處：

  • 思路更清晰：書寫過程能幫助您整理思緒。
  • 修改更容易：在 VS Code 中編輯文件，遠比在終端機中修改指令輕鬆。
  • 指令可複用：這份「簡報」可以存檔，下次有類似任務，改一改就能重複使用。

這個簡單的習慣，會讓您與 AI 的協作變得更穩定、更專業，產出品質也會更高。

本章總結

在本篇文章中，我們介紹了提升 PM 工作效率的實用 AI 技巧：

  • 利用 Gemini CLI 協作產生 Mermaid 流程圖，提升溝通效率。
  • 利用 Gemini CLI 聯網搜尋能力蒐集資料，為產品決策提供數據支持。
  • 建立 todo.md 任務清單，將複雜任務拆解，引導 AI 逐步完成。
  • 根據規格文件 (spec.md)，自動生成驗收測試案例，加速開發流程並確保品質。
  • 善用 Markdown 檔案預先管理複雜指令，提升溝通效率與穩定性。

透過這些 AI 驅動的策略，PM 可以將更多精力投入到策略思考和創新決策中，從「忙碌的管理者」轉型為「產品賦能者」，帶領團隊高效達成產品目標。
