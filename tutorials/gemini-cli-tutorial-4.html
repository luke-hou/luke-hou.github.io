PM x Gemini CLI (四)：AI 提升 PM 工作效率的進階實踐

發布日期：2025年10月25日

本篇文章將介紹能貫穿整個產品開發流程的實用 AI 技巧，提升溝通與分析效率。

文章大綱：

  • 要點一：利用 Gemini CLI 協作產生 Mermaid 流程圖
  • 要點二：利用 Gemini CLI 聯網搜尋能力蒐集資料
  • 要點五：用 Markdown 檔案準備「給 AI 的書面簡報」
  • 要點三：用 `todo.md` 導航複雜的產品開發流程
  • 要點四：讓 AI 自動生成「QA 的驗收清單」

要點一：用「說」的畫流程圖 (Mermaid 語法)

痛點場景： 為了跟工程師溝通一個複雜的用戶流程，您是否還在用 PowerPoint 或其他繪圖工具，辛苦地拉著線條、對齊著方塊？

現在，您可以換個方式：用「說」的畫流程圖。您只需用簡單的文字描述流程，AI 就能為您生成一種叫做 「Mermaid」的標準化圖表代碼。

範例：請 AI 生成「用戶登入流程圖」

您只要像平常一樣，條列出您的邏輯步驟：

請幫我生成一個 App 用戶登入的 Mermaid 流程圖，步驟如下：
1.  用戶打開 App
2.  檢查是否已登入
3.  如果已登入，就直接進主頁
4.  如果未登入，就顯示登入頁
5.  用戶輸入帳號密碼後點擊登入
6.  系統驗證帳密
7.  如果成功，進主頁
8.  如果失敗，顯示錯誤訊息並留在登入頁

Gemini CLI 就會產出對應的 Mermaid 代碼：

```mermaid
graph TD
    A[用戶打開 App] --> B{檢查是否已登入?};
    B -- 是 --> C[進入主頁];
    B -- 否 --> D[顯示登入頁面];
    D --> E[用戶輸入帳號密碼];
    E --> F[點擊登入按鈕];
    F --> G{驗證帳號密碼?};
    G -- 成功 --> C;
    G -- 失敗 --> H[顯示錯誤訊息];
    H --> D;
```

您可以將這段純文字的代碼，貼到任何支援 Mermaid 的地方 (例如 Mermaid.live、Notion、HackMD、VS Code 外掛等等)，它就會自動變成一張漂亮的流程圖。

並且您可以即時的在這些地方修改完畢後，再貼回本地的檔案內，節省一些跟 AI 來回交戰的時間與費用成本。

是的，這代表您的流程圖可以被輕鬆的修改、可以被輕易地複製分享，徹底告別過去的截圖地獄。

從此以後，當流程變動時，您再也不用重新拉線對齊，只需修改幾個字，就能產生新的流程圖，大幅提升溝通效率。

要點二：讓 AI 當您的「網路爬蟲」

痛點場景： 明天就要報告新功能的市場分析，您還在 Google 上手動搜尋、一篇篇點開文章、複製重點到筆記中？

Gemini CLI 內建了強大的聯網搜尋能力，能讓 AI 直接上網找資料，並根據您的要求進行即時的整理、分析、總結。它就是您最得力的情報官。

範例：10 分鐘完成競品功能分析

假設您想知道市場上的主流專案管理工具 (Asana, Jira, Trello) 是如何做「甘特圖」功能的，您可以這樣問：

你現在是一位資深的產品分析師。請幫我分析一下市面上三個主要專案管理工具 Asana, Jira, Trello 的「甘特圖」功能。

請直接上網搜尋最新資料，並從以下三個角度比較它們的優缺點：
1.  **易用性** (對非技術人員是否友善)
2.  **功能深度** (是否支援複雜的專案管理需求)
3.  **整合性** (與自身平台其他功能的結合程度)

最後，請用一個總結表格呈現你的分析。

Gemini CLI 模擬輸出：

好的，正在為您進行競品分析... 
GoogleSearch Searching the web for: "Asana vs Jira vs Trello Gantt chart features comparison 2025"

分析報告如下：

**主流專案管理工具甘特圖功能比較**

| 工具 | 易用性 | 功能深度 | 整合性 |
| :--- | :--- | :--- | :--- |
| **Asana** | ⭐⭐⭐⭐⭐ (非常直觀) | ⭐⭐⭐⭐ (中大型專案適用) | ⭐⭐⭐⭐⭐ (與任務無縫整合) |
| **Jira** | ⭐⭐ (較為複雜) | ⭐⭐⭐⭐⭐ (極度強大，適合開發) | ⭐⭐⭐⭐ (需透過插件整合) |
| **Trello**| ⭐⭐⭐⭐ (簡單好上手) | ⭐⭐ (適合小型專案) | ⭐⭐⭐ (需透過 Power-Up 整合) |

**總結觀察：**
- **Asana** 在易用性和功能深度上取得了最佳平衡，適合大部分混合型團隊。
- **Jira** 功能最專業、最深度，但學習曲線也最陡峭，是軟體開發團隊的首選。
- **Trello** 最輕量、最簡單，但功能也最受限，適合個人或小型團隊的簡單任務管理。

**給您的建議：** 如果您的目標用戶包含大量非技術人員 (如 PM、行銷、設計)，**Asana** 的設計會是最好的參考起點。

過去需要一小時的資料搜集與整理，現在 AI 在幾分鐘內就能完成，讓您能把時間專注在更高層次的策略判斷上。

範例：根據特定網頁內容優化文章

除了讓 AI 自由搜尋資料，您也可以指定一個網址，要求 AI 根據該網頁的內容來執行任務。這在您需要參考某篇專業文章、對標競品頁面，或是使用特定網頁來優化自己的內容時，特別有用。

假設您想讓 AI 根據一篇關於「敏捷開發」的部落格文章，來優化您自己撰寫的產品文件，使其更符合該文章的風格或論點：

你現在是一位專業的內容編輯。請閱讀這篇關於敏捷開發的文章：`https://example.com/agile-development-best-practices`。

然後，請根據這篇文章的風格、用詞和核心觀點，優化我提供的這份產品文件草稿 `@my_product_doc_draft.md`。

優化重點：
1.  確保術語使用一致。
2.  讓語氣更專業、更具說服力。
3.  如果文章中有任何可以借鑒的案例或論點，請適當融入。
4.  不要被人發現我在偷抄。

Gemini CLI 會先讀取您提供的網頁內容，然後再結合您的文件草稿進行優化。這讓 AI 的輸出更具針對性，也更能符合您的預期。

圖片說明：

圖片中，"紅色文字"代表 AI 幫您「刪除」的內容，而"綠色文字"則代表 AI 幫您「新增」的內容。這種視覺化的方式，能讓您一目瞭然地看到 AI 對文章的修改建議，方便您快速審閱和決策。

要點五：用 Markdown 檔案準備「給 AI 的書面簡報」

痛點場景： 當您有一個複雜的想法或任務，想在小小的終端機對話框裡一次打完，是不是常常覺得很難編輯、很容易按錯送出、或是事後很難找回當初完整的指令？

一個專業的工作習慣是：不要在對話框裡寫長篇大論。而是先把您的想法，有條理地寫在一個 Markdown (`.md`) 檔案裡。這對於處理類似「生成 PRD 初稿」等複雜任務時，尤其有用。

您可以把這個檔案，看作是一份要交給 AI 助理的「書面工作簡報 (Briefing)」。在裡面用標題、列表、粗體字，把背景資訊、任務要求、輸出格式等都寫清楚。

寫完後，只要在對話框中簡單地說：

請根據我這份簡報 `@my_complex_idea.md` 的內容，完成指定的任務。

這樣做的好處：

  • 思路更清晰：書寫過程能幫助您整理想法，釐清思緒，這是最最最重要的好處。
  • 修改更容易：在 VS Code 中編輯文件，遠比在終端機中修改指令輕鬆，並且可以直接看到 Markdown 渲染結果。
  • 指令可複用：這份「簡報」可以存檔，下次有類似任務，改一改就能重複使用。

文件範例：給 AI 的醫療資訊系統「Apache Score」表單規格書簡報 (`apache_score_spec.md`)

# 任務簡報：醫療資訊系統「Apache Score」表單規格書撰寫

## 任務背景
我們正在開發一套新的醫療資訊系統 (HIS)，其中需要實作「Apache Score」評分表單。此表單用於評估重症病患的生理狀況，並計算其預後分數。需要 AI 協助撰寫一份詳細的表單規格書，供前端開發與後端資料庫設計參考。

## 參考資料
*   **Apache Score 介紹**: (假設是) `https://zh.wikipedia.org/wiki/APACHE_II`
*   **相關醫學文獻**: (假設是) `https://www.ncbi.nlm.nih.gov/pmc/articles/PMC3078199/` (僅供參考，AI 無需實際讀取，主要理解其專業性)

## 任務要求（請按照順序執行）
1.  **理解 Apache Score 構成要素**：根據參考資料，列出 Apache Score 評分所需的所有生理參數、實驗室數據、年齡、慢性病史等構成要素。
2.  **表單欄位設計**：為每個構成要素設計對應的表單輸入欄位，包含：
    *   欄位名稱 (中文與英文代碼)
    *   資料型態 (例如：數值、下拉選單、日期)
    *   輸入限制 (例如：數值範圍、必填/非必填)
    *   預設值 (若有)
    *   單位 (若有)
    *   簡要說明 (用於前端提示或後端註解)
3.  **評分邏輯說明**：簡要說明 Apache Score 的計算邏輯，特別是各項參數如何轉換為分數。
4.  **特殊情境考量**：考量在醫療場景中可能遇到的特殊情境，例如：
    *   資料缺失時的處理方式。
    *   數值超出正常範圍時的提示。
    *   不同時間點數據的追蹤。

## 輸出格式
請以 Markdown 表格形式呈現表單欄位設計，並用清晰的標題和列表說明其他部分。

透過這樣詳細的「書面簡報」，AI 就能更精準地理解您的需求，產出高品質的成果。將這種協作方式融入您的日常工作流，將大大提升您的效率和精準度。

這個簡單的習慣，會讓您與 AI 的協作變得更穩定、更專業，產出品質也會更高。

這邊我們先了解這個概念，緊接著下一個步驟，就會說明如何用 todo.md 指揮 AI 自動完成 PRD 初稿。

要點三：用 `todo.md` 導航複雜的產品開發流程

痛點場景： 作為 PM，撰寫一份新的產品規格書 (PRD) 往往是一個漫長而複雜的過程，涉及大量的資料收集、整理、撰寫與原型驗證。如何確保每個環節都能有條不紊地進行，並最終產出高品質的 PRD 和 UI Prototype？

現在，您可以將 AI 視為您的專案助理，利用 `todo.md` 這個簡單而強大的概念，來導航整個 PRD 撰寫流程，確保每個步驟都清晰可控。

PM 撰寫 PRD 的 `todo.md` 工作流：

  1. 定義最終目標：PM 首先在 `todo.md` 中明確定義最終目標，例如「完成新功能 A 的 PRD 和 UI Prototype」。
  2. AI 協助拆解任務：PM 指示 AI 根據最終目標，將整個流程拆解為一系列具體、可執行的子任務，並列入 `todo.md`。
  3. PM 審核與指派：PM 審核 AI 拆解的任務清單，確認無誤後，逐一指派給不同的 AI Agent（例如：市場分析師、PM 分身、UI 設計師）或由 PM 親自執行。(這部分的操作會在之後的章節詳細說明。)
  4. AI Agent 執行與自動更新進度：每個 AI Agent 根據被指派的任務進行工作，並在完成後自動更新 `todo.md` 中的任務狀態（例如：從 `[ ]` 變為 `[x]`），PM 則負責審核每個階段的產出。

範例：撰寫新功能 PRD 的 `todo.md`

假設您需要為一個「智慧家居 App 的新安全監控功能」撰寫 PRD，您的 `todo.md` 可能會長這樣：

# 新功能 PRD 撰寫與 UI Prototype 任務清單：智慧家居安全監控

## 階段一：資料收集與市場分析
- [ ] **任務：** 進行競品分析，了解市場上主流安全監控 App 的功能與優勢。
    *   **指派給：** @market_analyst.md
    *   **驗收條件：** 產出競品分析報告 (`competitor_analysis.md`)。
- [ ] **任務：** 收集用戶對居家安全監控的需求與痛點。
    *   **指派給：** @user_researcher.md
    *   **驗收條件：** 產出用戶需求報告 (`user_needs_report.md`)。

## 階段二：PRD 初稿撰寫
- [ ] **任務：** 根據資料收集結果，撰寫新安全監控功能的 PRD 初稿。
    *   **指派給：** @pm_assistant.md
    *   **驗收條件：** 產出 PRD 初稿 (`security_feature_prd_draft.md`)，包含功能描述、用戶故事、成功指標。

## 階段三：UI Prototype 產出與驗證
- [ ] **任務：** 根據 PRD 初稿，設計安全監控儀表板的 UI 概念。
    *   **指派給：** @ui_designer.md
    *   **驗收條件：** 產出 UI 概念草圖描述 (`security_dashboard_ui_concept.md`)。
- [ ] **任務：** 根據 UI 概念，生成基於 Bootstrap 5 的 HTML UI Prototype。
    *   **指派給：** @frontend_dev_agent.md
    *   **驗收條件：** 產出可互動的 HTML UI Prototype (`security_dashboard_prototype.html`)。
- [ ] **任務：** 收集內部團隊對 UI Prototype 的回饋。
    *   **指派給：** PM (手動)
    *   **驗收條件：** 收集並整理回饋 (`prototype_feedback.md`)。

## 階段四：PRD 最終定稿
- [ ] **任務：** 根據 UI Prototype 回饋，優化 PRD 內容。
    *   **指派給：** @pm_assistant.md
    *   **驗收條件：** 產出最終版 PRD (`security_feature_prd_final.md`)。

這種以 `todo.md` 為核心的協作模式，讓 PM 能夠像一位總指揮，將複雜的產品開發流程拆解、指派、追蹤，確保每個環節都能高效且精準地完成，最終產出高品質的產品規格書與原型。

要點四：讓 AI 自動生成「功能驗收清單」

痛點場景： 您辛苦寫完的 PRD，交給開發團隊後，他們寫出來的功能，可能常因為雙方對需求規格的理解有偏差，而漏掉一些您在意的細節或邊界條件？

現在，您可以讓 AI 扮演 PM 和開發團隊之間的橋樑。在您寫完規格後，直接讓 AI 根據您的規格，產出第一版的「驗收測試案例 (Test Case)」。

範例：將「貼文發布」規格轉為測試案例

假設您的規格文件 spec.md 中有一段：

**功能：貼文發布**
- **使用者故事**：作為社群成員，我希望能發布一篇包含標題和內容的貼文。
- **驗收條件**：
    - 標題和內容都是必填。
    - 標題不可超過 100 字。
    - 成功發布後，應跳轉到新貼文頁面。

您可以這樣對 AI 下指令：

你是一位頂尖的 QA 專家。請根據我提供的規格文件 `@spec.md`，為「貼文發布」功能撰寫詳盡的驗收測試案例。

請務必包含：
- **正面案例** (正常操作)
- **負面案例** (錯誤操作)
- **邊界案例** (測試極端條件)

請用表格形式呈現，欄位包含「案例ID、測試項目、步驟、預期結果」。

**「貼文發布」功能驗收測試案例**

2. 頁面跳轉到新貼文 URL |

2. 輸入內容
3. 點擊「發布」 | 1. 提示「標題為必填」
2. 停在原頁面 |
| TC-003 | **負面**：內容為空 | 1. 輸入標題
2. 不輸入內容
3. 點擊「發布」 | 1. 提示「內容為必填」
2. 停在原頁面 |
| TC-004 | **邊界**：標題剛好100字 | 1. 輸入100字標題
2. 輸入內容
3. 點擊「發布」 | 1. 貼文成功

這份由 AI 生成的測試案例，可以作為您和 QA 團隊討論的基礎，確保每個人對需求的理解都在同一個水平上，從源頭提升產品質量。

  • 利用 Gemini CLI 協作產生 Mermaid 流程圖，提升溝通效率。
  • 利用 Gemini CLI 聯網搜尋能力蒐集資料，並根據特定網頁內容優化文章，為產品決策與內容創作提供數據支持。
  • 善用 Markdown 檔案準備「給 AI 的書面簡報」，有條理地管理複雜指令，提升溝通效率與穩定性。
  • 利用 `todo.md` 導航複雜的產品開發流程，從資料收集到 PRD 撰寫與原型產出，確保每個環節高效精準。
  • 讓 AI 自動生成「功能驗收清單」，加速開發流程並確保品質。

透過這些 AI 驅動的策略，PM 可以將更多精力投入到策略思考和創新決策中，從「忙碌的管理者」轉型為「產品賦能者」，帶領團隊高效達成產品目標。
